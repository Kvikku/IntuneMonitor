## Key Design Decisions

### Dual-Mode Entry Point

`Program.cs` checks `args.Length`:
- **No arguments** → Launches the interactive menu (`InteractiveMenu`)
- **With arguments** → Routes through `System.CommandLine` for direct CLI execution

Both paths share the same `AppConfiguration`, command classes, and logger factory.

### Storage Abstraction

`IBackupStorage` abstracts the backup backend:

```
IBackupStorage
├── LocalFileStorage     (writes JSON to timestamped directories)
├── GitStorage           (writes JSON + git commit + push)
└── AzureBlobStorage     (uploads JSON to Azure Blob Storage)
```

`BackupStorageFactory` selects the implementation based on `Backup.StorageType`.

### Content Type Registry

`IntuneContentTypes` is the central registry mapping each content type to:
- Its Microsoft Graph API endpoint
- Its JSON backup filename
- Its backup folder name

Adding a new content type only requires adding entries to these dictionaries.

### Comparison Engine

The comparison engine is split into focused classes:

- **`PolicyComparer`** — orchestrator that drives the comparison between live and backup data
- **`FieldComparer`** — computes field-level JSON differences between two policy snapshots
- **`AssignmentComparer`** — produces human-readable diffs for policy assignment arrays
- **`ChangeBuilder`** — factory for constructing `PolicyChange` instances with consistent severity assignment

The pipeline:

1. **Index** both sets by policy ID
2. **Added** = IDs in live but not backup
3. **Removed** = IDs in backup but not live
4. **Modified** = IDs in both, with deep JSON property comparison via `FieldComparer`

The comparer ignores volatile fields (`lastModifiedDateTime`, `version`, `@odata.type`) and handles assignment arrays with special human-readable formatting via `AssignmentComparer`.

### Reporting Pipeline

Each command can produce:
- **Console output** — structured logging via `ILogger` + styled tables via `ConsoleUI`
- **JSON reports** — serialized `ChangeReport` / `ExportReport` objects via `ReportWriter`
- **HTML reports** — self-contained dashboards generated by `HtmlReportGenerator` / `HtmlExportReportGenerator` / `HtmlAuditReportGenerator`, sharing utilities via `HtmlReportHelpers`
- **CSV reports** — tabular data via `CsvReportGenerator` with formula injection protection

## Extensibility

### Adding a New Content Type

1. Add a constant to `IntuneContentTypes`
2. Add entries to the `GraphEndpoints`, `FileNames`, and `FolderNames` dictionaries
3. That's it — export, import, monitor, and the interactive menu will pick it up automatically

### Adding a New Command

1. Create a new class in `Commands/` following the `ExportCommand` pattern (constructor takes `AppConfiguration` + `ILoggerFactory`)
2. Register the command in `CommandBuilder.cs`
3. Add a menu option constant in `MenuConstants.cs` and wire it in `InteractiveMenu.cs`
4. Update `docs/commands.md` and the commands table in `.github/copilot-instructions.md`

### Notification System

Drift alerts are sent via `INotificationSender` implementations:

- **`TeamsWebhookSender`** / **`SlackWebhookSender`** — extend `WebhookNotificationSender` base class
- **`EmailNotificationSender`** — sends via SMTP

`NotificationFactory` creates configured senders based on `AppConfiguration.Notifications`. `MonitorCommand` invokes them after drift detection.

## Testing

Tests live in `tests/IntuneMonitor.Tests/` and use xUnit (25 test files):

```bash
dotnet test
```

Tests cover the diff engine, Graph API clients (via mock `HttpMessageHandler`), commands, storage backends, notifications, configuration, and UI constants. Test helpers include `GraphTestHelpers` for creating fake `GraphClientFactory` instances and `MockHttpHandler` for stubbing HTTP responses.
